## Description

En phase de spécification puis de conception, se contraindre fonctionnellement et en termes d’UX avec les limitations d’un mobile (réseau aléatoire, petit écran, etc…) est recommandé. Lorsque cela est  justifié (par exemple par un fort gain de productivité comme pour une application d’emails), on peut envisager une déclinaison pour grands écrans. 

Ensuite, lors de l’implémentation, différentes mesures peuvent être mises en oeuvre, comme le chargement adaptatif, les media queries (notamment dans les attributs media des \<link> pour la sélection de feuilles de styles CSS), les attributs srcset et sizes des \<img>, les sous-éléments \<source> des \<picture>, \<video> et \<audio>. Il est aussi possible de récupérer ces informations via des API JavaScript pour éventuellement charger du code et/ou du contenu complémentaire dynamiquement. Utiliser la détection de user-agent côté serveur est fortement découragé par les acteurs du web.

Utiliser ces différentes techniques permet d’éviter de charger du contenu qui serait inutile ou inutilisable sur un terminal qui n’aurait pas les capacités requises.

## Exemple

Lors le la définition de l’application l’équipe a déterminé que toutes les fonctionnalités peuvent être implémentées en mobile-first. Néanmoins, deux fonctionnalités liées au tableau de bord et à la visualisation d’états pourraient être beaucoup plus confortables si elles exploitent pleinement un grand écran.

L’équipe a décidé d’utiliser une librairie javascript pour charger les écrans optimisés si l’utilisateur dispose d’une taille d’écran suffisante.
